# Anvil プロダクト機能分析

> **スコープ**: Anvilバイナリ（エンドユーザーが使うコーディングエージェント）の機能のみ。
> リポジトリの開発プロセス（CI/CD、Claude Codeスラッシュコマンド、オーケストレーション等）は対象外。

## 現状 (v0.0.2)

Anvilは **ローカルLLM特化のターミナルコーディングエージェント**。Ollama/OpenAI互換バックエンドに対応し、ファイル操作・シェル実行・Web検索等のツールを自律的に組み合わせてコーディングタスクを遂行する。

| 項目 | 現状 |
|------|------|
| ツール | file.read, file.write, file.edit, file.search, shell.exec, web.fetch, web.search + サブエージェント(explore/plan) |
| プロバイダー | Ollama, OpenAI互換API |
| 拡張機構 | MCP, Skills (SKILL.md), Hooks (4ライフサイクルポイント) |
| セキュリティ | サンドボックス, 危険コマンド検出, 権限クラス(Safe/Confirm/Restricted) |
| UI | ターミナルTUI, ストリーミング表示, y/n承認フロー |

---

## 1. 不足している機能

### エージェントとしての基本能力

| 機能 | 重要度 | なぜ必要か |
|------|--------|-----------|
| **undo/ロールバック** | 高 | file.write/file.editで壊した変更を戻す手段がない。checkpointはあるが復元操作がない。エージェントに「試行→失敗→戻す」のループをさせられない |
| **マルチファイルアトミック編集** | 中 | 関連する複数ファイルの変更を「全成功 or 全ロールバック」で行えない。途中でエラーが出ると中途半端な状態になる |
| **コンテキスト注入（@file, @url）** | 中 | ユーザーが「このファイルを見て」と指示する際、file.readをLLMに選ばせるしかない。明示的なコンテキスト注入が欲しい |

### ローカルLLM運用に必要な機能

| 機能 | 重要度 | なぜ必要か |
|------|--------|-----------|
| **モデル管理UI** | 中 | Ollamaで利用可能なモデル一覧の表示・切り替えがない。`/model list`, `/model switch` のようなコマンドが欲しい |
| **推論パフォーマンス表示** | 中 | tokens/sec, 推論時間がわからない。ローカルLLMではモデルサイズとハードウェアの相性が重要なのに判断材料がない |
| **オフラインモード** | 高 | web.fetch/web.searchがネットワーク必須。`--offline`で完全ローカル動作を保証するモードがない |

### ツールの不足

| 機能 | 重要度 | なぜ必要か |
|------|--------|-----------|
| **git専用ツール** | 中 | git操作は全てshell.exec経由。LLMがgitコマンド構文を正確に生成する必要があり、ローカルLLMでは精度が落ちやすい。git.diff, git.status等の専用ツールがあればLLMの負担を減らせる |
| **grep/ripgrep統合** | 中 | file.searchは基本的なファイル名/内容検索のみ。正規表現、除外パターン、コンテキスト行表示等のリッチな検索がない |

## 2. 改善すべき機能

### パフォーマンス

| 項目 | 現状の問題 | 改善案 |
|------|-----------|--------|
| **HTTPトランスポート** | 毎回curlプロセスを起動。ストリーミングでも新プロセス | reqwest等のネイティブHTTPクライアントに置き換え。接続プール、HTTP/2対応 |
| **トークン推定** | ヒューリスティック（文字数÷4 + CJK補正）。モデルごとのトークナイザーと乖離する | Ollama APIのトークン数レスポンスを活用。正確なカウントでコンテキスト枠を最大限利用 |
| **コンテキスト圧縮** | メッセージ数ベースの単純削除。重要な文脈が消える場合がある | LLMによるサマリー生成での圧縮。ツール実行結果の要約保持 |

### ユーザー体験

| 項目 | 現状の問題 | 改善案 |
|------|-----------|--------|
| **初回体験** | Ollama未起動時のエラーが技術的。モデル未ダウンロード時の案内なし | 「Ollamaが起動していません。`ollama serve`で起動してください」等のガイダンス |
| **承認フロー** | 毎回y/n。連続したfile.editで何度も確認が入る | 「このセッション中はfile.editを自動承認」等のバッチ承認。`--trust`モード |
| **セッション** | 1プロジェクト1セッション。前回の途中状態が常に残る | 名前付きセッション（`anvil --session bugfix-123`）、一覧・削除コマンド |
| **TUI** | スピナーとテキスト出力のみ。ツール実行状況が見づらい | 実行中ツール名・経過時間の表示改善。長いツール結果の折りたたみ |

### ローカルLLM特有の問題

| 項目 | 現状の問題 | 改善案 |
|------|-----------|--------|
| **ツール呼び出し精度** | ローカルLLMはツールJSON生成の精度が低い。repair_from_blockで補修しているが限界がある | ツール呼び出しプロトコルの簡素化。JSONではなくタグベース等、小モデルでも生成しやすい形式の検討 |
| **システムプロンプト肥大化** | 9ツール分の説明 + MCP + プロジェクト指示で巨大。ローカルLLMの限られたコンテキスト枠を圧迫 | 動的プロンプト生成。タスクに関連するツールのみ説明を含める |
| **コンテキスト枠の有効活用** | デフォルト4096トークン。ローカルLLMでは32K〜128K対応モデルも増えているが、活用しきれていない | モデル情報からcontext_windowを自動取得（Ollama API `/api/show`） |

## 3. 差別化になる機能

### 「ローカルLLMだからこそ」の強み

| 機能 | 競合との差 | 説明 |
|------|----------|------|
| **完全オフライン動作保証** | Claude Code/Cursor/Copilot にない | `--offline`モードでネットワークI/Oを完全遮断。機密コードが外に出ないことを技術的に保証。防衛・金融・医療等のセキュアな環境で唯一使えるコーディングエージェントになれる |
| **コスト$0の無制限利用** | クラウドAPI課金なし | ローカルGPUさえあれば利用制限なし。大量のイテレーション（TDD、リファクタリング）を気兼ねなく実行可能 |
| **モデルルーティング** | 独自 | 「file.searchはqwen3:8bで高速に、コード生成はqwen3:32bで高品質に」のようにタスク別モデル自動切り替え。ローカルに複数モデルを持てる利点を活かす |
| **推論プロファイリング** | 独自 | 各ターンのtokens/sec、レイテンシ、VRAM使用量をリアルタイム表示。「このモデルはこのマシンで実用的か」をユーザーが即座に判断できる |

### コーディングエージェントとしての差別化

| 機能 | 競合との差 | 説明 |
|------|----------|------|
| **RAG統合（ローカルベクトルDB）** | ローカル完結 | sqlite-vss等でコードベースをインデックス化。ローカルembeddingモデルでベクトル生成。プライバシーを保ったまま大規模コードベースのセマンティック検索が可能 |
| **学習モード（判断過程の可視化）** | 独自 | 「なぜこのツールを選んだか」「なぜこの編集をしたか」をステップごとに表示。ローカル実行なのでレイテンシを気にせず詳細ログを出せる。教育用途にも |
| **プロジェクト適応** | 独自 | プロジェクトのコーディング規約・パターンをANVIL.mdで定義→システムプロンプトに注入する仕組みはあるが、さらにembeddingで類似コードを自動参照すれば精度が上がる |

### エコシステム

| 機能 | 競合との差 | 説明 |
|------|----------|------|
| **IDE統合** | 必須 | VSCode/Neovim拡張。ターミナルだけでなくエディタ内からAnvilのツールを使える。ユーザー層が大幅に広がる |
| **モデルベンチマーク** | 独自 | `anvil benchmark`でプロジェクト固有のタスク（ファイル読み→編集→テスト）を各モデルで実行し、精度・速度を比較。最適モデルを推薦 |

## 優先度マトリクス

```
                    実装コスト 低 ─────────────────────── 高
                    │                                     │
                    │ ★ オフラインモード                   │ ★ RAG統合
 ユーザー ─────── │ ★ undo/ロールバック                  │ ★ IDE統合
 インパクト高      │ ★ 初回体験/エラーUX改善             │ ★ モデルルーティング
                    │ ★ 推論パフォーマンス表示            │
                    │ ★ セッション管理改善                │
                    │ ★ バッチ承認                        │
                    │                                     │
 ユーザー ─────── │   モデル管理UI                       │   プラグインシステム
 インパクト低      │   コンテキスト注入(@file)            │   モデルベンチマーク
                    │   git専用ツール                     │   学習モード
                    │                                     │
```

**最優先**: オフラインモード、undo、初回体験改善は実装コスト低×インパクト高。ローカルLLMエージェントとしてのアイデンティティ確立に直結する。

## 実装ロードマップ

セクション1（不足機能）とセクション2（改善機能）の全項目を、**共通基盤の共有**と**依存関係**に基づき4つのWaveに整理した。各Wave内は並列実装可能。Wave間は前のWaveの成果を利用する。

### 依存関係の全体像

```
Wave 1 (基盤整備)
  ├── オフラインモード ─────────────────────────────────┐
  ├── 初回体験/エラーUX ──→ Ollama API活用 ──┐         │
  ├── コンテキスト枠自動取得 ────────────────┤         │
  └── 推論パフォーマンス表示 ────────────────┘         │
                                                       │
Wave 2 (エージェント中核)                               │
  ├── undo/ロールバック ──→ マルチファイルアトミック編集  │
  ├── バッチ承認 (--trust)                              │
  └── セッション管理改善                                │
                                                       │
Wave 3 (ツール・プロトコル)                             │
  ├── ツール呼び出しプロトコル簡素化 ──┐                │
  ├── システムプロンプト動的生成 ──────┤                │
  ├── grep/ripgrep統合 ───────────────┤                │
  ├── git専用ツール ───────────────────┤                │
  ├── コンテキスト注入 (@file) ────────┘                │
  └── モデル管理UI ──────────────────────── Wave 1に依存 ┘

Wave 4 (パフォーマンス)
  ├── ネイティブHTTPクライアント
  ├── トークン推定精度向上 ──→ スマートコンテキスト圧縮
  └── TUI改善
```

---

### Wave 1: 基盤整備 — ローカルLLMの「使い始め」を改善

> **テーマ**: provider層の強化。Ollama APIとの連携を深め、ローカルLLMならではの情報を引き出す。
> **共通基盤**: `src/provider/ollama.rs` と `src/config/mod.rs` の拡張

| # | 機能 | 変更対象 | 概要 |
|---|------|---------|------|
| W1-1 | **初回体験/エラーUX改善** | provider/ollama.rs, provider/mod.rs | Ollama未起動・モデル未DL時にユーザーフレンドリーなガイダンスを表示。ヘルスチェック結果のメッセージ改善 |
| W1-2 | **コンテキスト枠自動取得** | provider/ollama.rs, config/mod.rs | Ollama `/api/show` からモデルのcontext_lengthを取得し、config.context_windowに自動設定。手動設定不要に |
| W1-3 | **推論パフォーマンス表示** | provider/mod.rs, tui/mod.rs, contracts/mod.rs | Ollamaレスポンスのeval_count/eval_durationからtokens/secを算出。TUIのステータス行に表示 |
| W1-4 | **オフラインモード** | config/mod.rs, tooling/mod.rs | `--offline`フラグ追加。有効時はweb.fetch/web.searchをToolExecutionStatus::Failedで即返却。ネットワークI/Oゼロを保証 |

**Wave 1の効果**: ユーザーが初めてAnvilを起動した瞬間から「ローカルLLM向けに作られている」と感じられる。

---

### Wave 2: エージェント中核 — 「安心して任せられる」ための安全機構

> **テーマ**: ファイル変更の安全網とセッション管理。エージェントに大胆な変更をさせるための信頼基盤。
> **共通基盤**: `src/session/mod.rs` と `src/app/mod.rs` の拡張

| # | 機能 | 変更対象 | 概要 |
|---|------|---------|------|
| W2-1 | **undo/ロールバック** | tooling/mod.rs, app/mod.rs, session/mod.rs | 既存のcheckpointBeforeWriteを活用し、`/undo`コマンドで直前のファイル変更を復元。チェックポイントのスタック管理を追加 |
| W2-2 | **マルチファイルアトミック編集** | tooling/mod.rs, app/agentic.rs | W2-1のcheckpointスタックを利用。ツールグループ単位でチェックポイント→全実行→失敗時ロールバック |
| W2-3 | **バッチ承認** | app/agentic.rs, app/cli.rs, config/mod.rs | `--trust`フラグおよび`/trust file.edit`コマンドで、セッション中の特定ツールを自動承認。承認状態をセッションに記録 |
| W2-4 | **セッション管理改善** | session/mod.rs, config/mod.rs, app/cli.rs | `--session <name>`で名前付きセッション。`/session list`, `/session switch`, `/session delete`コマンド。セッションファイルのディレクトリ構造変更 |

**依存関係**: W2-2はW2-1のcheckpoint機構に依存。W2-3とW2-4は独立して実装可能。

**Wave 2の効果**: 「undoがあるから安心」→ユーザーが`--trust`で全承認→エージェントの自律実行が実用的に。

---

### Wave 3: ツール・プロトコル — ローカルLLMでの精度と効率を上げる

> **テーマ**: ローカルLLM（特に小モデル）がツールを正しく使えるようにする。プロトコル簡素化とツール追加。
> **共通基盤**: `src/tooling/mod.rs` と `src/agent/mod.rs` の拡張

| # | 機能 | 変更対象 | 概要 |
|---|------|---------|------|
| W3-1 | **ツール呼び出しプロトコル簡素化** | agent/mod.rs, tooling/mod.rs | ANVIL_TOOLブロック形式の代替として、タグベース形式（`<tool name="file.read" path="./src/main.rs"/>`）を追加。小モデルでもJSON構文エラーが起きにくい形式。既存JSON形式との併存 |
| W3-2 | **システムプロンプト動的生成** | agent/mod.rs | 全ツール説明を常に含めるのではなく、直近の会話で使われたツール＋基本ツールのみ説明。MCP/サブエージェントの説明は初回使用後に追加。プロンプトサイズを30-50%削減 |
| W3-3 | **grep/ripgrep統合** | tooling/mod.rs | file.searchを拡張し、正規表現検索・除外パターン（.gitignore準拠）・コンテキスト行表示をサポート。ripgrepバイナリがあれば利用、なければRust内蔵のregex |
| W3-4 | **git専用ツール** | tooling/mod.rs | git.status, git.diff, git.log を専用ツールとして追加。LLMがgitコマンド構文を知らなくても使える。shell.exec経由より安全（引数バリデーション付き） |
| W3-5 | **コンテキスト注入 (@file)** | app/cli.rs, app/mod.rs | ユーザー入力の`@path/to/file`を検出し、ファイル内容をユーザーメッセージに自動展開。LLMにfile.readを選ばせる1ターンを省略 |
| W3-6 | **モデル管理UI** | extensions/mod.rs, provider/ollama.rs | `/model list`（Ollama `/api/tags`）、`/model switch <name>`（セッション中のモデル切替）、`/model info`（パラメータ数、量子化、コンテキスト長の表示）。Wave 1のOllama API活用基盤に依存 |

**依存関係**: W3-1とW3-2は連携が大きい（プロンプト内のツール説明形式がプロトコルと一致する必要がある）。W3-3〜W3-6は独立。W3-6はWave 1のOllama API基盤に依存。

**Wave 3の効果**: 8Bクラスの小モデルでもツール呼び出し成功率が向上。開発者の日常操作（grep, git）がエージェント経由で快適に。

---

### Wave 4: パフォーマンス — 内部品質の底上げ

> **テーマ**: ユーザーに見える変化は少ないが、体感速度と安定性が向上する。
> **共通基盤**: `src/provider/transport.rs` と `src/contracts/tokens.rs` のリプレース

| # | 機能 | 変更対象 | 概要 |
|---|------|---------|------|
| W4-1 | **ネイティブHTTPクライアント** | provider/transport.rs, Cargo.toml | curlサブプロセスをreqwestに置き換え。接続プール・HTTP/2・ストリーミングのネイティブ対応。毎ターンのプロセス起動コスト（~5ms）を削減 |
| W4-2 | **トークン推定精度向上** | contracts/tokens.rs, provider/mod.rs | Ollama/OpenAIレスポンスの`prompt_eval_count`/`completion_tokens`を蓄積し、ヒューリスティックを補正。モデル別の補正係数を学習 |
| W4-3 | **スマートコンテキスト圧縮** | session/mod.rs, app/mod.rs | メッセージ数ではなくトークン数ベースで圧縮判定。ツール実行結果を要約に置き換え（「file.readで200行読み取り→要約: config構造体の定義を含むファイル」）。W4-2の正確なトークン数に依存 |
| W4-4 | **TUI改善** | tui/mod.rs, spinner.rs | 実行中ツール名・経過時間のリアルタイム表示。長いツール結果の折りたたみ。Wave 1の推論パフォーマンス表示と統合 |

**依存関係**: W4-3はW4-2に依存（正確なトークン数がないとスマート圧縮の判定が不正確）。W4-1とW4-4は独立。

**Wave 4の効果**: 体感レスポンスが改善。コンテキスト枠を賢く使い、長いセッションでも品質が落ちにくくなる。
